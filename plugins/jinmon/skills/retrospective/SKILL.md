## いつ使うか

- jinmonのステップ6として自動的に呼ばれるとき
- 「今回のNGをrulesに反映して」と指示されたとき
- タスク完了後に同じ失敗を繰り返したくないと感じたとき

## 実行手順

### 1. 会話を振り返る

今回のタスクでNGが発生したか確認する。NGとは以下のいずれかが起きた状態:

- 人間がテストリストをNG → 修正して再提示した
- 人間が動作確認でNG → 実装をやり直した
- /code-review または /security-review で問題が検出された
- 人間から「これは違う」「やり直して」などのフィードバックがあった

NGがなかった場合はその旨を伝えて終了する。

### 2. NGを分類・整理する

NGごとに以下を整理する:

```
NG内容: <何がNGだったか>
原因タイプ:
  - A(スタイル): コーディングスタイル・設計パターンのズレ
  - B(スコープ): 要求していないことをやった
  - C(ドメイン): プロジェクト固有の制約を知らなかった
再発可能性: 高 / 中 / 低
```

再発可能性が低いもの(一度きりの状況判断ミスなど)はルール化しない。

### 3. ルールを生成して提案する

再発可能性が中・高のNGに対して、ルールの追加案を生成して人間に提示する。

**提示フォーマット:**

```
今回のNGから以下のルールを追加することを提案します。

---
【NG】<何がNGだったか>
【原因】<A/B/C + 一言説明>
【提案するルール】
  <具体的なルール文>
【追加先】<ファイル名>
---

追加しますか？ (yes / no / 修正して追加)
```

### 4. 承認されたルールをファイルに書き込む

**追加先の判断:**

| 内容 | 追加先 |
|------|--------|
| コーディングスタイル・設計 | `coding-philosophy.md` |
| テストの書き方 | `testing-strategy.md` |
| PRの作り方 | `pull-request.md` |
| プロジェクト固有の制約 | プロジェクトの `CLAUDE.md` |

人間が「修正して追加」を選んだ場合は `/grill-me` の要領で修正内容が明確になるまで確認してから書き込む。

### 5. 完了報告

```
追加したルール: <件数>件
スキップしたNG: <件数>件(再発可能性低のため)
```
