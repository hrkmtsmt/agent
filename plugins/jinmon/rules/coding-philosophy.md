# コーディング哲学

## ネスト

- if文のネストは1階層まで
- 条件分岐は早期リターンで解決すること

## 純粋性

- 副作用のない純粋関数・メソッドを実装すること

## 再代入

- 変数・オブジェクトのプロパティへの再代入は禁止する
- 値はイミュータブルに扱うこと

## コードの重複

- 同じ処理が3回以上繰り返される場合は関数・メソッドに切り出す
- 例外として、同じ並びの処理でも目的が異なる場合は共通化不要(密結合を避けるため)

## コメント

- コードの意図・理由を説明するために使用すること
- コードの内容をそのまま説明するコメントは禁止する
- 例外として、複雑なロジックでコメントなしでは理解が困難な場合は内容説明も許可

## Deep Module / Shallow Module

- モジュール・関数・クラスは深くあること(John Ousterhout "A Philosophy of Software Design")
- Deep module(良い): シンプルなインターフェースの裏に大きな機能を持っていて、呼び出し側の複雑性を吸収する
- Shallow module(悪い): インターフェースと実装の複雑性がほぼ同じで、呼び出し側に複雑性を押し付ける

別の関数を呼ぶだけ・プロパティにアクセスするだけのShallow moduleは禁止。

## 単一責任の原則

関数・クラス・モジュールはそれぞれ単一のアクターに対して責任を持つこと(アンクル・ボブの定義に従う)。

## 実装の粒度

- 1つのPRに構造的変更(リファクタリング)と振る舞いの変更(機能追加・バグ修正)を混在させないこと
- リファクタリングと機能追加は別々のPRに分けること
