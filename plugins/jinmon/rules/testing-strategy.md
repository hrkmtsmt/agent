# テスト戦略

## テストの分類

- テストの分類はGoogleが提唱するテストサイズ(Small / Medium / Large)を用いること(ユーザー間で用語の定義がずれないようにするため)
- ユニットテスト、インテグレーションテスト、E2Eテストという表現は使わないこと(テストトロフィーやテストピラミッドで使われるこれらの表現はユーザーによって解釈が分かれるため)
- もしこれらの表現をユーザーが使った場合は以下のように解釈すること
  - ユニットテスト -> Small Test
  - インテグレーションテスト -> Medium Test
  - E2Eテスト -> Large Test

Test Sizes: https://testing.googleblog.com/2010/12/test-sizes.html

## テスト方針

- 1つのテストケースに対して1つのアサーションしか書いてはならない(アサーションルーレットを避けるため)
- テストを実装するときはテストダブルは使わず本物を使うこと
  - 例外1: テスト対象のコードがWeb APIやファイルシステムなど、別のプロセスで動く外部サービスに依存している場合はテストダブルを使うこと
  - 例外2: テスト対象のコードがDBに依存している場合は、テスト用のDBコンテナを立ててテストを実行すること(プロダクトで使われているDBは使わないこと)
- テストファイルはテスト対象のファイルと同じディレクトリに配置すること(コロケーション)

## テストのスコープ

- テストは振る舞いに対して行うこと
- 実装の詳細をテストしてはいけない
- プライベートメソッドやモジュールとして公開されていない関数のテストは行わないこと
- テストのためにプライベートなものを公開してはいけない

## テストケースの表現

- テストケースの表現はGiven When Thenパターンに習うこと
- テストケースは仕様レベルで表現すること
- テストケースに実装の詳細を表現してはいけない
- テストケースはプログラマ以外の人間も理解できる表現にすること

Introducing BDD: https://dannorth.net/blog/introducing-bdd/
Given When Then: https://martinfowler.com/bliki/GivenWhenThen.html

Bad（実装の詳細を表現している）: 「calculateTax が呼ばれること」「status フィールドが 'active' にセットされること」
Good（Given-When-Then パターンで仕様レベルで表現している）: 「割引クーポンを適用したとき、税込み価格が割引後の金額で計算される」「メールアドレスを確認したとき、ユーザーが有効化される」

## テストケースの数

- テストケースの数は必要最低限に抑えること(コードが増えると保守性が低くなるため)
- テストケースには同値分割と境界分析を用いること

## テストケースの構造

- クラスをテストする場合、3段構造を基本とする(対象クラス -> メソッド -> テスト)
- 関数をテストする場合、2段構造を基本とする(対象関数 -> テスト)
- 関数のテストで前提条件や状態によって振る舞いが大きく変わる場合のみ、3段構造を許容する(対象関数 -> 前提条件・状態 -> テスト)
- ネスト構造をサポートしていない言語・フレームワークの場合は、`[対象]_[メソッド or 状態]_[期待する振る舞い]` の形式の命名規則でネスト構造を表現すること
- ネストは最大3段までとする

## 実装前のテストリスト

- 実装を開始する前に、テストケースのリストを作成すること
- テストリストはテストフレームワークの未実装マーカー機能を使ってテストファイルに記述すること
- テストリストは人間がレビューしてOKを出してから実装を開始すること
- テストリストの各項目はGiven-When-Thenパターンで仕様レベルで表現すること(実装の詳細を含めないこと)

テストリストの例（ユーザー登録）:
- 有効なメールと名前を入力したとき、ユーザーが作成される
- 既存のメールアドレスで登録したとき、エラーが返る
- メールアドレスの形式が不正なとき、バリデーションエラーが返る
- 名前が空のとき、バリデーションエラーが返る
