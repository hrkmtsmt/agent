# エスカレーション基準

Agentは以下のいずれかに該当する状況では、独断で進めず**必ず人間に確認してから**次のアクションを取ること。

## エスカレーションすべき状況

### 1. 要件・仕様の曖昧さ

- 仕様に複数の解釈が存在し、どちらを選ぶかによって設計・実装が大きく変わる
- 受け入れ要件に記載されていないエッジケースを発見した
- `/grill-me` で合意した仕様と、実装中に判明した制約が矛盾している

### 2. スコープの逸脱

- タスクを完了するために、合意したスコープ外のファイル・モジュールの変更が必要になった
- 1つのTODOテストを実装しようとしたら、他のテストに影響が出ることが分かった

### 3. 実リソースへの不可逆な操作

コード上の変更のみであればエスカレーション不要。以下のように実際のリソースを変更する場合のみ対象とする。

- 本番DBのデータ削除・上書き・マイグレーション実行
- 外部サービスへの書き込み・送信（APIコール、メール送信など）
- デプロイ・インフラ変更の実行

### 4. 競合する制約

- `coding-philosophy.md` と `CLAUDE.md` のルールが矛盾している
- パフォーマンス・セキュリティ・可読性のトレードオフで正解が一意に定まらない

### 5. 未知のドメイン

- `CLAUDE.md` に記載のないアーキテクチャ上の判断が必要になった
- プロジェクトで初めて使うライブラリ・パターンを導入しようとしている

## エスカレーションの形式

確認が必要な場合は以下の形式で人間に伝えること。

```
【確認が必要です】

状況: <何が起きているか>
判断が必要な理由: <なぜ独断で進められないか>
選択肢:
  A: <選択肢Aの説明と影響>
  B: <選択肢Bの説明と影響>
推奨: <AまたはBと、その理由>

どちらで進めますか？
```

## エスカレーションしてはいけない状況

以下は確認不要。独断で進めること。

- `coding-philosophy.md` に明記されている規約の適用
- Lint・フォーマットの自動修正
- テストが通っているかどうかの確認
- `CLAUDE.md` に記載された制約の範囲内での実装判断
